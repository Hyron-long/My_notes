# 第五章 运输层

## 5.1 运输层协议概述

### 5.1.1 进程之间的通信

运输层在计算机网络体系结构中处于关键位置，它位于**应用层**和**网络层**之间：

- **网络层**：提供主机之间的逻辑通信（端到端）
- **运输层**：提供**进程之间**的逻辑通信（端到端）

**核心理解**：
- IP协议虽然能把分组送到目的主机，但这个分组还停留在主机的网络层，没有交付给主机中的应用进程
- 运输层的作用就是**将数据交付给正确的进程**，即实现**复用**和**分用**

**复用与分用**：
| 概念     | 方向   | 说明                                       |
| -------- | ------ | ------------------------------------------ |
| **复用** | 发送方 | 多个应用层进程可同时使用同一个运输层协议   |
| **分用** | 接收方 | 运输层将收到的数据正确交付给对应的应用进程 |

### 5.1.2 运输层的两个主要协议

TCP/IP协议族的运输层包含两个主要协议：

| 协议    | 全称           | 类型     | 特点                                 |
| ------- | -------------- | -------- | ------------------------------------ |
| **UDP** | 用户数据报协议 | 无连接   | 简单、不可靠、面向报文               |
| **TCP** | 传输控制协议   | 面向连接 | 可靠、面向字节流、流量控制、拥塞控制 |

> **注意**：在TCP/IP协议族中，运输层使用**端口号**来区分不同的应用进程。

### 5.1.3 运输层的端口

**端口（Port）** 是运输层服务访问点（TSAP），用于标识主机中的应用进程。

**端口号的范围**（16位，0~65535）：

| 端口范围      | 类型                            | 说明                                                   |
| ------------- | ------------------------------- | ------------------------------------------------------ |
| 0 ~ 1023      | **熟知端口**（Well-known Port） | 分配给全球通用的重要应用，如HTTP(80)、FTP(21)、DNS(53) |
| 1024 ~ 49151  | **登记端口**                    | 需要在IANA登记，避免重复使用                           |
| 49152 ~ 65535 | **短暂端口**（动态端口）        | 客户进程临时使用，通信结束后释放                       |

**常用熟知端口**（需要记忆）：

| 端口号  | 协议    | 应用             |
| ------- | ------- | ---------------- |
| 20      | TCP     | FTP数据传送      |
| 21      | TCP     | FTP控制连接      |
| 22      | TCP     | SSH              |
| 23      | TCP     | Telnet           |
| 25      | TCP     | SMTP（邮件发送） |
| 53      | UDP/TCP | DNS              |
| 67/68   | UDP     | DHCP             |
| 69      | UDP     | TFTP             |
| 80      | TCP     | HTTP             |
| 110     | TCP     | POP3（邮件接收） |
| 111     | TCP/UDP | RPC              |
| 135-139 | TCP/UDP | NetBIOS          |
| 143     | TCP     | IMAP             |
| 161     | UDP     | SNMP             |
| 443     | TCP     | HTTPS            |
| 520     | UDP     | RIP              |
| 1080    | TCP     | SOCKS            |

## 5.2 用户数据报协议UDP

### 5.2.1 UDP概述

**UDP（User Datagram Protocol）** 只在IP数据报服务之上增加了很少的一点功能：**复用和分用**以及**差错检测**。

**UDP的主要特点**：

1. **无连接**：发送数据前不需要建立连接，减少了开销和时延
2. **不可靠**：不保证可靠交付，不使用确认机制
3. **面向报文**：应用层交给UDP的报文，UDP直接添加首部后交给IP层，不拆分也不合并
   - 即：UDP对应用层交下来的报文，既不合并也不拆分，保留报文边界
4. **无拥塞控制**：网络拥塞不会降低UDP的发送速率（适用于实时应用）
5. **支持一对一、一对多、多对一和多对多交互通信**
6. **首部开销小**：首部只有8字节

**UDP的适用场景**：
- 实时性要求高但可靠性要求不高的应用：视频流、VoIP、在线游戏
- 需要组播或多播的应用
- 简单的请求-响应协议：DNS、SNMP、DHCP

### 5.2.2 UDP的首部格式

UDP首部固定为**8字节**：

| 16位源端口号     | 16位目的端口号 |
| ---------------- | -------------- |
| 16位UDP长度      | 16位UDP检验和  |
| 数据部分（可变） |                |

| 字段           | 长度 | 说明                                |
| -------------- | ---- | ----------------------------------- |
| **源端口号**   | 16位 | 发送方端口号，不需要时可填0         |
| **目的端口号** | 16位 | 接收方端口号，必须填写              |
| **UDP长度**    | 16位 | UDP首部+数据的总长度（字节），最小8 |
| **UDP检验和**  | 16位 | 检测UDP用户数据报在传输中是否有错   |
| **数据**       | 可变 | 应用层数据                          |

**UDP检验和的计算**：
- UDP的检验和**既检验首部，也检验数据**
- 计算时需要加上**伪首部**（仅用于计算检验和，不发送）：
  - 伪首部包含：源IP地址（4B）、目的IP地址（4B）、0（1B）、协议号(17)（1B）、UDP长度（2B）
- 检验和数据部分不是偶数时，填充一个全0字节

## 5.3 传输控制协议TCP

### 5.3.1 TCP的主要特点

**TCP（Transmission Control Protocol）** 是面向连接的、可靠的、基于字节流的运输层协议。

**TCP的主要特点**：

1. **面向连接**：数据传输前必须先建立连接，数据传输结束后释放连接
2. **点对点**：每条TCP连接只能有两个端点（一对一）
3. **可靠交付**：无差错、不丢失、不重复、按序到达
4. **全双工通信**：允许双方随时发送数据
5. **面向字节流**：TCP将应用层交下来的数据看成无结构的字节流，不保留报文边界
6. **流量控制**：使用滑动窗口机制防止发送方发送过快
7. **拥塞控制**：根据网络拥塞程度调整发送速率

**TCP连接的定义**：TCP连接是**虚连接**或**逻辑连接**，不是真正的物理连接。TCP连接的端点是**套接字（Socket）**。

### 5.3.2 TCP的连接端点——套接字

**套接字（Socket）** = IP地址 + 端口号

**每条TCP连接唯一地被通信两端的两个套接字标识**：

> **TCP连接 ::= { socket1, socket2 } = { (IP1:port1), (IP2:port2) }**

> **注意**：这里的Socket与编程接口中的Socket概念不同，但密切相关。

### 5.3.3 TCP报文段的首部格式

TCP报文段由**首部**和**数据**两部分组成。TCP首部最小为**20字节**，最大为60字节。

**TCP首部格式（固定部分20字节）**：

| 16位源端口号                 | 16位目的端口号 |
| ---------------------------- | -------------- |
| 32位序号                     |                |
| 32位确认号                   |                |
| 4位数据偏移                  | 6位保留        |
| 16位检验和                   | 16位紧急指针   |
| 选项（长度可变，最多40字节） | 填充           |

**各字段详细说明**：

| 字段              | 长度 | 说明                                                         |
| ----------------- | ---- | ------------------------------------------------------------ |
| **源端口号**      | 16位 | 发送方端口号                                                 |
| **目的端口号**    | 16位 | 接收方端口号                                                 |
| **序号（seq）**   | 32位 | 本报文段所发送数据的第一个字节的序号                         |
| **确认号（ack）** | 32位 | 期望收到对方下一个报文段的第一个数据字节的序号；**ACK=1时才有效** |
| **数据偏移**      | 4位  | TCP首部长度，单位是4字节（最小5=20B，最大15=60B）            |
| **保留**          | 6位  | 保留为今后使用，应为0                                        |
| **URG**           | 1位  | 紧急指针有效（紧急数据立即发送，不排队）                     |
| **ACK**           | 1位  | 确认号有效（连接建立后所有报文段ACK必须为1）                 |
| **PSH**           | 1位  | 推送（接收方应尽快将数据交付给应用进程，不等待缓存满）       |
| **RST**           | 1位  | 重置连接（严重错误时释放连接，重新建立）                     |
| **SYN**           | 1位  | 同步序号（用于连接建立）                                     |
| **FIN**           | 1位  | 终止连接（发送方完成数据发送）                               |
| **窗口**          | 16位 | 接收窗口（rwnd），告诉对方自己的接收能力，范围0~65535        |
| **检验和**        | 16位 | 检验首部和数据（计算时需加上12字节的TCP伪首部）              |
| **紧急指针**      | 16位 | URG=1时有效，指出紧急数据的末尾位置                          |
| **选项**          | 可变 | 常用的有：MSS选项、窗口扩大选项、时间戳选项、SACK选项等      |
| **填充**          | 可变 | 确保首部长度是4字节的整数倍                                  |

**TCP标志位速记**：**"URG ACK PSH RST SYN FIN"**（考试常考）

| 标志 | 含义           | 用途                 |
| ---- | -------------- | -------------------- |
| SYN  | Synchronize    | 同步序号，建立连接   |
| ACK  | Acknowledgment | 确认，表示确认号有效 |
| FIN  | Finish         | 终止，释放连接       |
| RST  | Reset          | 重置，异常释放       |
| URG  | Urgent         | 紧急，配合紧急指针   |
| PSH  | Push           | 推送，立即交付       |

**关于序号和确认号的重要说明**：
- TCP是**面向字节流**的，**序号**是字节的序号，不是报文段的序号
- 例如：一个报文段的序号为301，数据共200字节，则表明本报文段数据的第一个字节序号是301，最后一个字节序号是500
- 此时若收到确认号501，表示序号500及之前的数据都已收到，期望收到序号501开始的报文段

## 5.4 可靠传输的工作原理

TCP的可靠传输基于以下机制：**序号**、**确认**、**超时重传**。

### 5.4.1 停止等待协议

**停止等待协议**是最简单的可靠传输协议：发送方每发送一个分组，就停止发送，等待对方的确认；收到确认后再发送下一个分组。

**可能出现的三种情况**：

#### 1. 无差错情况
- 发送分组 → 接收方确认 → 发送方收到确认 → 发送下一个分组

#### 2. 出现差错（超时重传）
- 发送分组 → 接收方检测到差错（丢弃）→ 发送方超时未收到确认 → 重传分组
- **超时计时器**：每发送一个分组就启动，重传时间应比平均往返时间RTT稍长

#### 3. 确认丢失和确认迟到
- **确认丢失**：接收方已发送确认，但确认丢失在途中 → 发送方超时重传 → 接收方收到重复分组，丢弃重复分组，并**重新确认**
- **确认迟到**：确认在重传后到达 → 发送方收到重复确认，丢弃

**停止等待协议的缺点**：信道利用率太低，发送方大部分时间都在等待确认。

### 5.4.2 连续ARQ协议

**连续ARQ（Automatic Repeat reQuest）** 使用**流水线传输**和**滑动窗口**，大幅提高信道利用率。

**工作原理**：
- 发送方维持一个**发送窗口**，窗口内的字节可以连续发送，无需等待确认
- 接收方采用**累积确认**：收到几个分组后，只对按序到达的最后一个分组发送确认
  - 例如：确认号n表示序号n之前的所有字节都已正确收到
- 发送方收到确认后，窗口向前滑动

**Go-back-N（回退N步）**：
- 当发送方发现某个分组丢失或出错（超时未收到确认）时，必须**重传该分组及其之后的所有分组**
- 即使后续分组已经收到了确认，也要重传（因为接收方收到失序分组不会确认，只暂存或丢弃）

**连续ARQ的优缺点**：
| 优点         | 缺点                                              |
| ------------ | ------------------------------------------------- |
| 信道利用率高 | Go-back-N可能浪费带宽（重传多个已正确收到的分组） |
| 实现相对简单 | 累积确认导致不能及时反馈单个分组丢失              |

> **TCP实际使用的是**：**滑动窗口 + 选择确认（SACK）** 的改进版本，可以只重传丢失的分组，而不是回退N步。

## 5.5 TCP的流量控制

### 5.5.1 流量控制的基本概念

**流量控制**：让发送方的发送速率不要太快，使接收方来得及接收。

TCP使用**滑动窗口机制**实现流量控制：

- 接收方在TCP首部中的**窗口字段**（rwnd，receiver window）告诉发送方自己还有多少接收缓存空间
- 发送方的发送窗口不能超过接收方给出的接收窗口大小

**窗口的协同工作**：
```
发送窗口 = min(接收方rwnd, 拥塞窗口cwnd)
```

### 5.5.2 持续计时器和零窗口探测

**零窗口**：当接收方通知窗口为0时，发送方停止发送。

**持续计时器（Persistent Timer）**：
- 当发送方收到零窗口通知后，启动持续计时器
- 计时器到期后，发送一个**零窗口探测报文段**
- 接收方收到探测报文后，回复当前的窗口值
- 如果窗口还是0，持续计时器再次启动（时间加倍）

### 5.5.3 糊涂窗口综合征（Silly Window Syndrome）

**糊涂窗口综合征**：当接收方每次只释放很小的窗口，发送方每次只发送很小的数据，导致网络效率低下。

**解决方法**：
- **接收方**：不通告很小的窗口值（通常当窗口小于最大报文段长度MSS或1/2缓存时不通告）
- **发送方**：不发送太小的报文段（使用Nagle算法）

**Nagle算法**：
- 若发送数据较小，先缓存，等到累计到足够大（如MSS）或收到之前数据的确认后再发送
- 适用于广域网，在局域网中通常禁用（如交互式应用）

## 5.6 TCP的拥塞控制

### 5.6.1 拥塞控制的一般原理

**拥塞（Congestion）**：网络中对资源的需求超过了可用资源，导致网络性能下降的现象。

**拥塞控制与流量控制的区别**：

| 控制类型     | 目的                                   | 关注点               |
| ------------ | -------------------------------------- | -------------------- |
| **流量控制** | 防止发送方发送过快导致接收方来不及接收 | 端到端（接收方能力） |
| **拥塞控制** | 防止过多的数据注入网络，避免网络过载   | 全网（网络容量）     |

**拥塞控制的代价**：为了进行拥塞控制，需要获得网络拥塞的信息，这会带来额外的开销。

### 5.6.2 TCP拥塞控制算法

TCP的拥塞控制由四个核心算法组成：**慢开始**、**拥塞避免**、**快重传**、**快恢复**。

#### 基本概念

| 参数         | 含义              | 说明                                 |
| ------------ | ----------------- | ------------------------------------ |
| **cwnd**     | 拥塞窗口          | 发送方根据网络拥塞程度动态调整的窗口 |
| **rwnd**     | 接收窗口          | 接收方通告的窗口（接收能力）         |
| **发送窗口** | = min(cwnd, rwnd) | 实际发送窗口                         |
| **ssthresh** | 慢开始门限        | 用于切换慢开始和拥塞避免的阈值       |

#### 1. 慢开始（Slow Start）

**慢开始原则**：从小到大逐渐增大拥塞窗口。

- 刚开始发送时，cwnd = 1（或2）个MSS
- 每收到一个ACK，cwnd翻倍（指数增长）
- cwnd达到ssthresh时，进入拥塞避免阶段

> "慢开始"并不意味着cwnd增长慢，而是指从1开始逐渐增加，而不是一开始就发送大量数据。

#### 2. 拥塞避免（Congestion Avoidance）

**拥塞避免原则**：让cwnd线性增长（加法增大）。

- 每经过一个RTT，cwnd增加1个MSS（而不是翻倍）
- 当检测到拥塞（超时）时：
  - ssthresh = max(cwnd/2, 2)
  - cwnd = 1
  - 重新进入慢开始阶段

**拥塞判断**：通过**超时重传**判断拥塞的发生。

#### 3. 快重传（Fast Retransmit）

**快重传机制**：当发送方连续收到**三个重复ACK**时，立即重传丢失的报文段，而不必等到超时。

- 接收方收到失序报文段时，立即发出重复ACK（确认序号不变）
- 发送方收到3个重复ACK → 认为该报文段已丢失 → 立即重传
- 快重传可以避免发送方因超时而过早进入慢开始（缩短恢复时间）

#### 4. 快恢复（Fast Recovery）

**快恢复机制**：与快重传配合使用，当使用快重传时，不进入慢开始，而是进入拥塞避免。

- ssthresh = cwnd / 2
- cwnd = ssthresh
- 直接进入拥塞避免阶段（加法增大）

> 注意：超时重传使用慢开始 + 拥塞避免；快重传使用快恢复 + 拥塞避免。

### 5.6.3 拥塞控制的完整流程

```
发送方
  │
  ▼
 慢开始（指数增长）
  │
  ▼ cwnd == ssthresh
 拥塞避免（线性增长）
  │
  ├─── 超时（拥塞）─────────────────→ ssthresh = cwnd/2, cwnd=1
  │
  └─── 收到3个重复ACK（快重传）───→ ssthresh = cwnd/2, cwnd = ssthresh
         │
         └────────────────────────→ 进入拥塞避免（快恢复）
```

### 5.6.4 主动队列管理AQM

**主动队列管理（Active Queue Management, AQM）** 是路由器层面的拥塞控制辅助机制。

**RED（Random Early Detection，随机早期检测）**：
- 当路由器队列长度超过一定阈值时，以一定概率丢弃新到达的分组
- 通过提前丢包通知发送方网络可能发生拥塞
- 避免全局同步问题（多个TCP流同时进入慢开始）

## 5.7 TCP的运输连接管理

TCP是面向连接的协议，运输连接有三个阶段：**连接建立** → **数据传送** → **连接释放**。

### 5.7.1 TCP的连接建立（三次握手）

TCP使用**三次握手（Three-way Handshake）** 建立连接。

#### 正常的三次握手过程：

```
客户端 A                                    服务器 B
    │                                           │
    │  ① SYN=1, seq=x                           │
    │──────────────────────────────────────────>│
    │                                           │
    │  ② SYN=1, ACK=1, seq=y, ack=x+1           │
    │<──────────────────────────────────────────│
    │                                           │
    │  ③ ACK=1, seq=x+1, ack=y+1                │
    │──────────────────────────────────────────>│
    │                                           │
```

**各步骤详解**：

| 步骤           | 报文                         | 说明                                                       |
| -------------- | ---------------------------- | ---------------------------------------------------------- |
| **第一次握手** | SYN=1, seq=x                 | 客户端请求建立连接，进入SYN_SENT状态                       |
| **第二次握手** | SYN=1, ACK=1, seq=y, ack=x+1 | 服务器确认客户端的SYN，同时发送自己的SYN，进入SYN_RCVD状态 |
| **第三次握手** | ACK=1, seq=x+1, ack=y+1      | 客户端确认服务器的SYN，进入ESTABLISHED状态，可传送数据     |

> **问题**：为什么需要三次握手，而不是两次？
>
> 防止**已失效的连接请求报文段**突然又传到服务器，导致错误建立连接。
>
> 场景：客户端第一次发送的SYN延迟了（已超时重传），后来到达服务器。服务器回复SYN+ACK后，若只有两次握手，连接就建立了。但实际上客户端并没有请求连接，造成服务器资源浪费。三次握手中，客户端不发送第三次ACK，服务器收不到确认，就不会建立连接。

### 5.7.2 SYN洪泛攻击

**SYN洪泛攻击**：攻击者发送大量SYN报文，但不完成第三次握手，使服务器处于SYN_RCVD状态，占满连接队列，导致无法响应正常请求。

**防御措施**：
- SYN Cookie技术：服务器收到SYN时，不立即分配资源，而是根据源IP、端口等信息计算一个cookie作为seq返回
- 缩短SYN_TIMEOUT时间
- 设置最大半连接数

### 5.7.3 TCP的连接释放（四次挥手）

TCP使用**四次挥手（Four-way Wavehand）** 释放连接。

#### 正常的四次挥手过程：

```
客户端 A                                    服务器 B
    │                                           │
    │  ① FIN=1, seq=u                           │
    │──────────────────────────────────────────>│  A: FIN-WAIT-1
    │                                           │  B: CLOSE-WAIT
    │  ② ACK=1, seq=v, ack=u+1                  │
    │<──────────────────────────────────────────│
    │                                           │
    │  ③ FIN=1, ACK=1, seq=w, ack=u+1           │
    │<──────────────────────────────────────────│  B: LAST-ACK
    │                                           │
    │  ④ ACK=1, seq=u+1, ack=w+1                │
    │──────────────────────────────────────────>│  A: TIME-WAIT
    │                                           │
    │  （等待2MSL后关闭）                        │  B: CLOSED
```

**各步骤详解**：

| 步骤           | 报文           | 说明                                                         |
| -------------- | -------------- | ------------------------------------------------------------ |
| **第一次挥手** | FIN=1, seq=u   | 客户端主动关闭，进入FIN-WAIT-1状态                           |
| **第二次挥手** | ACK=1, ack=u+1 | 服务器确认收到FIN，进入CLOSE-WAIT状态；客户端收到后进入FIN-WAIT-2 |
| **第三次挥手** | FIN=1, seq=w   | 服务器关闭连接，进入LAST-ACK状态                             |
| **第四次挥手** | ACK=1, ack=w+1 | 客户端确认服务器的FIN，进入TIME-WAIT状态；等待2MSL后关闭     |

**为什么需要四次挥手？**
- TCP是全双工的，需要双方分别释放连接
- 第一次挥手是A告诉B"我发完了"，B确认
- 第二次握手后，B可能还有数据要发送（半关闭状态）
- 第三次挥手是B告诉A"我也发完了"
- 第四次挥手是A确认

**为什么需要TIME-WAIT状态（等待2MSL）？**

**MSL（Maximum Segment Lifetime）**：报文段最大生存时间。

原因：
1. 保证最后一个ACK能到达服务器（若丢失，服务器会重发FIN，客户端可以重发ACK）
2. 防止已失效的连接请求报文段出现在本连接中

> **2MSL**：等待时间为2倍的MSL（RFC建议为2分钟，常用30秒、1分钟或2分钟）

### 5.7.4 TCP的有限状态机

TCP连接的各种状态及其转换可以用**有限状态机（FSM）** 表示。

**主要状态**：

| 状态        | 含义                            |
| ----------- | ------------------------------- |
| CLOSED      | 关闭状态，没有连接              |
| LISTEN      | 服务器等待连接请求              |
| SYN_SENT    | 客户端已发送SYN，等待确认       |
| SYN_RCVD    | 服务器收到SYN，已发送SYN+ACK    |
| ESTABLISHED | 连接已建立，可传输数据          |
| FIN-WAIT-1  | 客户端发送FIN，等待ACK          |
| FIN-WAIT-2  | 客户端收到FIN的ACK，等待对方FIN |
| CLOSE-WAIT  | 服务器收到FIN，等待应用程序关闭 |
| LAST-ACK    | 服务器发送FIN，等待ACK          |
| TIME-WAIT   | 客户端收到FIN的ACK，等待2MSL    |
| CLOSING     | 双方同时发送FIN时进入此状态     |

### UDP vs TCP 对比

| 特性     | UDP           | TCP                 |
| -------- | ------------- | ------------------- |
| 连接     | 无连接        | 面向连接            |
| 可靠性   | 不可靠        | 可靠                |
| 面向     | 报文          | 字节流              |
| 拥塞控制 | 无            | 有                  |
| 首部大小 | 8字节         | 20~60字节           |
| 适用     | 实时应用、DNS | 文件传输、Web、邮件 |

### TCP首部重要字段

> 源端口(16) + 目的端口(16) + seq(32) + ack(32) + 偏移(4) + 保留(6) + **URG,ACK,PSH,RST,SYN,FIN(6)** + 窗口(16) + 检验和(16) + 紧急指针(16)

### 三次握手（建立连接）

> **SYN → SYN+ACK → ACK**

### 四次挥手（释放连接）

> **FIN → ACK → FIN → ACK**

### 拥塞控制四算法

> **慢开始**（指数增长）→ **拥塞避免**（线性增长）→ **快重传**（3个重复ACK立即重传）→ **快恢复**（不进入慢开始）

### 重要公式/数值

- **发送窗口** = min(cwnd, rwnd)
- **ssthresh** = max(拥塞时cwnd/2, 2)
- **MSL** = 报文段最大生存时间
- **TIME-WAIT** = 2 × MSL

### 常见TCP状态转换

```
CLOSED → LISTEN → SYN_RCVD → ESTABLISHED → FIN-WAIT-1 → FIN-WAIT-2 → TIME-WAIT → CLOSED
         SYN_SENT ↗
```
