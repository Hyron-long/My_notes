## 2. 核心知识点

### 2.1 死锁的概念

**死锁（Deadlock）：** 一组相互竞争系统资源或进行通信的进程间相互"永久"阻塞的状态，每个进程都在等待只能由该组中另一个进程才能引发的事件。

**死锁的四个必要条件（Coffman条件，1971）：**

1. **互斥（Mutual Exclusion）：** 资源一次只能被一个进程使用
2. **持有并等待（Hold and Wait）：** 进程已持有资源，同时等待其他资源
3. **不可抢占（No Preemption）：** 资源只能由持有者自愿释放
4. **循环等待（Circular Wait）：** 存在进程等待环路 P1→P2→...→Pn→P1

**四个条件必须同时满足才会发生死锁。**

### 2.2 死锁的处理策略

```
       死锁处理
       /  |  \
    预防  避免  检测与恢复
```

#### 2.2.1 死锁预防（Deadlock Prevention）

破坏四个必要条件之一：

| 破坏的条件 | 方法 | 缺点 |
|-----------|------|------|
| 互斥 | 共享所有资源 | 有些资源本质不可共享 |
| 持有并等待 | 进程开始前一次性申请全部资源 | 资源利用率低，可能饥饿 |
| 不可抢占 | 抢占已分配的资源 | 实现复杂，可能丢失工作 |
| 循环等待 | 对所有资源排序，按序申请 | 资源排序困难，限制灵活性 |

#### 2.2.2 死锁避免（Deadlock Avoidance）

**核心思想：** 允许三个必要条件，但在资源分配时进行判断，确保不会进入死锁状态。

**需要的信息：**
- 每个进程的最大资源需求（事先声明）
- 当前已分配的资源
- 当前可用的资源

**安全状态：** 存在一个安全序列（P1, P2, ..., Pn），按此顺序分配资源不会导致死锁。

**银行家算法（Banker's Algorithm）：**
```
数据结构：
- Available[m]: 每种资源可用数量
- Max[n][m]: 每个进程的最大需求
- Allocation[n][m]: 每个进程已分配的资源
- Need[n][m]: 每个进程还需要的资源 = Max - Allocation

安全检查算法：
1. 令 Work = Available, Finish[i] = false
2. 找到一个满足 Finish[i] == false 且 Need[i] <= Work 的i
3. Work = Work + Allocation[i], Finish[i] = true
4. 重复2-3直到所有 Finish[i] == true（安全）或找不到（不安全）

资源请求算法（请求i进程的资源 Request[]）：
1. 如果 Request > Need[i]，拒绝（超出声明）
2. 如果 Request > Available，等待
3. 假装分配：Available -= Request, Allocation[i] += Request, Need[i] -= Request
4. 运行安全检查
5. 安全→正式分配；不安全→恢复，进程等待
```

**实例：**
```
系统有3类资源 A(10), B(5), C(7)
Available = [3, 3, 2]

      Allocation  Max     Need
      A B C      A B C   A B C
P0    0 1 0     7 5 3   7 4 3
P1    2 0 0     3 2 2   1 2 2
P2    3 0 2     9 0 2   6 0 0
P3    2 1 1     2 2 2   0 1 1
P4    0 0 2     4 3 3   4 3 1

安全序列：P1 → P3 → P4 → P0 → P2 ✓ 系统处于安全状态
```

#### 2.2.3 死锁检测与恢复

**检测时机：**
- 每次资源请求时检测（频繁但早期发现）
- 定时检测（减少开销但延迟）
- 资源利用率下降时检测

**检测方法：**
1. 构建资源分配图
2. 使用类似银行家算法的检测算法
3. 寻找图中的环路

**恢复方法：**
| 方法 | 描述 | 代价 |
|------|------|------|
| **终止进程** | 终止所有死锁进程，或逐个终止直到死锁解除 | 丢失计算成果 |
| **资源抢占** | 从某些进程抢占资源分配给其他进程 | 需要回滚和补偿 |

**选择终止进程的考虑因素：**
- 优先级
- 已运行时间和还需运行时间
- 已使用资源数量和类型
- 还需资源数量
- 进程类型（批处理 vs 交互）

### 2.3 哲学家就餐问题

```
     P4       P0
   🥢     🍝     🥢
  P3      桌      P1
   🥢     🍝     🥢
     P2
```

五个哲学家围坐，五根筷子。每人需要左右两根筷子才能吃饭。

**死锁场景：** 所有哲学家同时拿起左边的筷子，然后等待右边的筷子。

**解决方案：**
1. 最多允许四个哲学家同时就座
2. 奇数号先拿左筷，偶数号先拿右筷（打破循环等待）
3. 相邻两哲学家只能一个就餐
4. 使用信号量保护拿筷子的操作

### 2.4 饥饿（Starvation）

**定义：** 进程长时间等待但理论上最终可以获得资源（与死锁不同，死锁是永久的）。

**与死锁的区别：**
| 特性 | 死锁 | 饥饿 |
|------|------|------|
| 等待性质 | 永久等待 | 可能最终获得资源 |
| 发生条件 | 四个必要条件同时满足 | 资源分配策略不公平 |
| 解决方法 | 预防/避免/检测恢复 | 公平调度（如FCFS） |

## 3. 重要术语

| 英文术语 | 中文翻译 | 说明 |
|---------|---------|------|
| Deadlock | 死锁 | 进程间永久性相互阻塞 |
| Starvation | 饥饿 | 进程长期不能获得所需资源 |
| Livelock | 活锁 | 进程不断改变状态但无实际进展 |

| Safe State | 安全状态 | 存在安全完成序列的系统状态 |
| Banker's Algorithm | 银行家算法 | Dijkstra提出的死锁避免算法 |
| Resource Allocation Graph | 资源分配图 | 用图论分析死锁的工具 |

## 5. 关键要点总结

- **死锁必须同时满足四个条件**：互斥、持有并等待、不可抢占、循环等待
- **死锁预防**：破坏四个必要条件之一（最简单但资源利用率低）
- **死锁避免**：利用银行家算法判断分配后是否安全（需预知最大需求）
- **死锁检测与恢复**：不预防不死锁，检测到后再恢复（适用于死锁不频繁的系统）
- **银行家算法**的核心是判断是否存在安全序列
- **饥饿**不同于死锁：死锁是永久的，饥饿可能最终获得资源
- 实际系统中常用混合策略：重要资源用**有序分配法**，临时资源用**超时机制**
