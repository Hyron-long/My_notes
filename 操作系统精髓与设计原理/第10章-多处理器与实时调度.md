## 2. 核心知识点

### 2.1 多处理器系统的分类

| 类型 | 描述 | 例子 |
|------|------|------|
| **松耦合多处理器** | 多个独立处理器，通过通信链路连接 | 集群系统 |
| **紧耦合多处理器** | 多个处理器共享同一内存 | SMP系统 |
| **多核处理器** | 单个物理芯片上集成多个处理器核心 | 现代CPU |

**多核系统的层次：**
```
┌─────────────── 芯片(CPU) ───────────────┐
│  ┌──── 核心1 ────┐  ┌──── 核心2 ────┐  │
│  │ L1 Cache      │  │ L1 Cache      │  │
│  │ L2 Cache      │  │ L2 Cache      │  │
│  └───────────────┘  └───────────────┘  │
│  ┌────────────────────────────────────┐  │
│  │         共享 L3 Cache / 内存        │  │
│  └────────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### 2.2 多处理器调度的设计问题

1. **同步粒度（Granularity）：**
   - 细粒度并行：指令级并行（需要频繁同步）
   - 中粒度并行：线程级并行
   - 粗粒度并行：进程级并行（同步开销小）
   - 独立并行：完全独立的进程（无需同步）
   - 无约束并行：不需要同步

2. **亲和性（Processor Affinity）：**
   - **软亲和性：** 尽量在同一处理器上运行（不强制）
   - **硬亲和性：** 明确指定在哪个处理器上运行
   - 优势：利用Cache预热数据，减少Cache Miss

3. **负载均衡（Load Balancing）：**
   - **Push迁移：** 定期检查，将任务从忙处理器移到闲处理器
   - **Pull迁移：** 空闲处理器从忙处理器拉取任务
   - 与亲和性的冲突：负载均衡可能需要迁移进程，降低Cache命中率

### 2.3 多处理器调度方法

#### 负载共享（Load Sharing）
- 所有处理器共享一个全局就绪队列
- 空闲处理器从队列中取下一个任务
- 优点：简单，负载自然均衡
- 缺点：队列成为瓶颈（需加锁），失去亲和性

#### 组调度（Gang Scheduling）
- 将一组相关的线程同时调度到不同的处理器上
- 同时启动，同时切换
- 适合紧密协作的并行应用
- 避免线程间忙等

#### 专用处理器分配（Dedicated Processor Assignment）
- 每个线程分配一个专用处理器
- 处理器不共享，不需要切换
- 适合处理器数量充足的情况
- 可能造成处理器浪费

#### 动态调度（Dynamic Scheduling）
- 由应用程序和操作系统共同决定
- 应用程序告知OS并行度变化
- OS调整处理器分配

### 2.4 Linux多核调度

**完全公平调度器（CFS — Completely Fair Scheduler）：**
- 红黑树组织所有可运行进程
- 虚拟运行时间（vruntime）作为键值
- 选择vruntime最小的进程运行
- 自动在多核间负载均衡

**负载均衡：**
- 定期检查各CPU的运行队列长度
- 从不均衡的CPU迁移任务到空闲CPU
- 考虑Cache亲和性和NUMA拓扑

### 2.5 实时系统

**定义：** 系统的正确性不仅取决于计算结果，还取决于结果产生的时间。

**实时系统分类：**
| 类型 | 描述 | 例子 |
|------|------|------|
| **硬实时** | 必须在截止时间前完成，否则系统失败 | 飞行控制、汽车ABS |
| **软实时** | 偶尔错过截止时间可以容忍，性能下降 | 视频播放、VoIP |

**实时操作系统的特点：**
- 确定性的调度延迟
- 快速中断响应
- 优先级倒置处理
- 高分辨率定时器

### 2.6 实时调度算法

#### 单调速率调度（RMS — Rate Monotonic Scheduling）
- 静态优先级分配
- 周期越短（频率越高）的进程优先级越高
- 优先级在运行时不改变
- 可抢占

**可调度性条件：**
```
对于n个周期任务，保证可调度的条件：
Σ(Ci/Ti) ≤ n(2^(1/n) - 1)

Ci: 执行时间, Ti: 周期

n=2: 利用率上限 ≈ 83%
n→∞: 利用率上限 → ln2 ≈ 69%
```

#### 最早截止时间优先（EDF — Earliest Deadline First）
- 动态优先级分配
- 截止时间最早的任务优先级最高
- **最优的**单处理器实时调度算法
- 可达到100% CPU利用率

**RMS vs EDF：**
| 特性 | RMS | EDF |
|------|-----|-----|
| 优先级 | 静态 | 动态 |
| 实现复杂度 | 简单 | 较复杂 |
| 最大CPU利用率 | ~69% | 100% |
| 过载行为 | 可预测（低优先级先丢失） | 难以预测 |
| 稳定性 | 稳定 | 可能不稳定 |

#### 优先级反转（Priority Inversion）

**问题：**
```
高优先级进程H → 等待低优先级进程L释放锁
中等优先级进程M → 抢占L运行
结果：H被M间接阻塞（M优先级低于H）
```

**解决方案 — 优先级继承（Priority Inheritance）：**
- L持有锁时临时获得等待它的最高优先级
- 防止中等优先级进程抢占

**解决方案 — 优先级天花板（Priority Ceiling）：**
- 每把锁预设一个优先级天花板（所有可能请求该锁的任务中的最高优先级）
- 持有锁的进程自动继承天花板优先级

## 3. 重要术语

| 英文术语 | 中文翻译 | 说明 |
|---------|---------|------|
| SMP | 对称多处理器 | 多个相同处理器共享内存 |
| NUMA | 非一致内存访问 | 不同内存区域访问速度不同 |
| Processor Affinity | 处理器亲和性 | 进程倾向于在同一处理器上运行 |
| Load Balancing | 负载均衡 | 在多个处理器间均匀分配负载 |
| Gang Scheduling | 组调度 | 同时调度一组相关线程 |
| Hard Real-Time | 硬实时 | 必须满足所有截止时间 |
| RMS | 单调速率调度 | 按周期分配优先级的实时调度 |
| EDF | 最早截止时间优先 | 按截止时间动态分配优先级 |
| Priority Inversion | 优先级反转 | 高优先级进程被低优先级间接阻塞 |
| Priority Inheritance | 优先级继承 | 解决优先级反转的方案 |
| CFS | 完全公平调度器 | Linux的默认调度器 |
| Tardiness | 延迟 | 任务完成时间超出截止时间的量 |

## 5. 关键要点总结

- 多处理器调度需处理**亲和性vs负载均衡**的权衡
- **负载共享**简单但队列成为瓶颈；**组调度**适合紧密协作的并行应用
- **RMS**是静态优先级调度，利用率上限约69%
- **EDF**是动态优先级调度，可达100%利用率且理论上最优
- **优先级反转**是实时系统中的常见问题，通过**优先级继承**解决
- 实时操作系统的关键：**确定性、快速中断响应、避免优先级反转**
- Linux的**CFS**使用红黑树和虚拟运行时间实现公平调度
