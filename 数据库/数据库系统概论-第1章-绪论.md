# 第1章 绪论

## 1.1 数据库系统概述

### 1.1.1 数据库的4个基本概念

1. **数据（Data）**：描述事物的符号记录。数据有多种表现形式（数字、文本、图形、图像、音频、视频等），都可以经过数字化后存入计算机。数据的语义与数据本身不可分割。

2. **数据库（Database, DB）**：长期存储在计算机内有组织、可共享的大量数据的集合。数据按一定的数据模型组织、描述和存储，具有较小的数据冗余、较高的数据独立性和可扩展性。

3. **数据库管理系统（DBMS）**：位于用户与操作系统之间的数据管理软件。主要功能包括：
   - 数据定义功能（DDL）
   - 数据组织、存储和管理功能
   - 数据操纵功能（DML）
   - 数据库的事务管理和运行管理功能
   - 数据库的建立和维护功能
   - 其他功能（通信、数据转换、异构互访等）

4. **数据库系统（DBS）**：引入数据库后的计算机系统，由数据库、DBMS（及其应用开发工具）、应用系统和数据库管理员（DBA）组成。

### 1.1.2 数据管理技术的产生和发展

数据管理技术经历了三个阶段：

| 对比项 | 人工管理阶段 | 文件系统阶段 | 数据库系统阶段 |
|--------|-------------|-------------|---------------|
| 应用领域 | 科学计算 | 科学计算、数据管理 | 大规模数据管理 |
| 主要硬件 | 无直接存取存储设备 | 磁盘、磁鼓 | 大容量磁盘、磁盘阵列 |
| 主要软件 | 无OS、无管理数据软件 | 有文件系统 | 有DBMS |
| 数据处理方式 | 批处理 | 联机实时处理、批处理 | 联机实时处理、分布处理、批处理 |
| 数据管理者 | 人（程序员） | 文件系统 | DBMS |
| 数据面向对象 | 某一应用程序 | 某一应用 | 现实世界（部门、企业等） |
| 数据共享程度 | 不共享，冗余度极高 | 共享性弱，冗余度高 | 共享性强，冗余度低且易扩充 |
| 数据独立性 | 不独立 | 独立性弱 | 物理独立性+逻辑独立性 |
| 数据结构化 | 无结构 | 记录内有结构、整体无结构 | 整体结构化，用数据模型描述 |
| 数据控制能力 | 应用程序自己控制 | 应用程序自己控制 | DBMS提供安全、完整、并发、恢复 |

**数据库系统阶段的四个特点：**
1. **整体数据的结构化**（本质区别）
2. **数据的共享性强、冗余度低且易于扩充**
3. **数据的独立性强**（物理独立性 + 逻辑独立性）
4. **数据由DBMS统一管理和控制**
   - 安全性保护（Security）
   - 完整性检查（Integrity）
   - 并发控制（Concurrency）
   - 数据库恢复（Recovery）

## 1.2 数据模型

数据模型是数据库系统的核心和基础，是对现实世界数据特征的抽象。

### 1.2.1 数据建模

两步抽象过程：
1. **现实世界 → 信息世界**：建立概念模型（按用户观点建模）
2. **信息世界 → 机器世界**：转换为DBMS支持的数据模型（按计算机系统观点建模）

### 1.2.2 概念模型

信息世界的基本概念：
- **实体（Entity）**：客观存在并可相互区别的事物
- **属性（Attribute）**：实体所具有的某一特性
- **码（Key）**：唯一标识实体的属性集
- **实体型（Entity Type）**：实体名 + 属性名集合
- **实体集（Entity Set）**：同一类型实体的集合
- **联系（Relationship）**：实体之间的联系（一对一、一对多、多对多）

**E-R模型（实体-联系模型）**：由P.P.S. Chen于1976年提出，用E-R图描述概念模型。

### 1.2.3 数据模型的三要素

1. **数据结构**：描述数据库的组成对象及对象之间的联系（静态特性）
2. **数据操纵**：对数据库中对象允许执行的操作集合（动态特性）
3. **完整性约束**：数据的制约和依存规则，保证数据的正确、有效和相容

### 1.2.4 层次模型

- 树形结构表示实体及实体间联系
- 两个条件：①有且只有一个根结点 ②根以外结点有且只有一个双亲结点
- 典型代表：IBM IMS（1968年）
- 优点：结构简单清晰、查询效率高、完整性约束良好
- 缺点：不能直接表示多对多联系、查询必须通过双亲节点、程序化命令

### 1.2.5 网状模型

- 两个条件：①允许多个结点无双亲 ②结点可以有多个双亲
- 典型代表：DBTG/CODASYL系统
- 优点：更直接描述现实世界、存取效率高
- 缺点：结构复杂、DDL/DML复杂、不便于用户使用

### 1.2.6 关系模型

- 1970年由Edgar F. Codd提出（1981年图灵奖）
- 数据结构：二维表
- 关键术语：
  - **关系**：一张二维表
  - **元组**：表中的一行
  - **属性**：表中的一列
  - **码**：唯一确定一个元组的属性组
  - **域**：属性的取值范围
  - **分量**：元组中的一个属性值
  - **关系模式**：关系名(属性1, 属性2, ..., 属性n)
- 要求关系必须是规范化的（每个分量是不可分的数据项，不允许表中有表）
- 数据操纵是集合操作，存取路径对用户隐蔽
- 优点：严格的数学基础、概念单一、独立性强、安全保密性好
- 缺点：查询效率不如层次/网状模型，需要查询优化

### 1.2.7 新型数据模型

- 面向对象数据模型
- 对象关系数据模型
- 半结构化XML数据模型
- 键值对（KV）数据模型
- 文档数据模型、图数据模型、时序数据模型、时空数据模型等

## 1.3 数据库系统的三级模式结构

### 1.3.1 模式的概念

- **型（Type）**：对某一类数据的结构和属性的说明
- **值（Value）**：型的一个具体赋值
- **模式（Schema）**：数据库中全体数据的逻辑结构和特征的描述（只涉及型的描述）
- **实例（Instance）**：模式的一个具体值（反映数据库某一时刻的状态）

### 1.3.2 三级模式结构

1. **模式（Schema）**：数据库数据在逻辑级上的视图。一个数据库只有一个模式，不涉及物理存储。

2. **外模式（External Schema）**：也称子模式/用户模式，是用户可见的局部数据的逻辑结构和特征的描述。一个数据库可以有多个外模式，是保证数据库安全性的有力措施。

3. **内模式（Internal Schema）**：也称存储模式，是对数据物理结构和存储方式的描述。一个数据库只有一个内模式。

### 1.3.3 两级映像与数据独立性

1. **外模式/模式映像**：定义外模式与模式之间的对应关系。模式改变时，修改映像使外模式不变 → 保证**数据逻辑独立性**

2. **模式/内模式映像**：定义全局逻辑结构与存储结构之间的对应关系。存储结构改变时，修改映像使模式不变 → 保证**数据物理独立性**

> **核心结论**：两级映像保证了数据库外模式的稳定性，从底层保证了应用程序的稳定性。

## 1.4 数据库系统的组成

1. **硬件平台**：大容量存储设备与高速处理器
2. **软件平台**：OS、DBMS、高级语言及编译系统、应用开发工具、数据库应用系统
3. **人员**：
   - **DBA**：设计与定义数据库、帮助用户使用、运维、性能调优、转储与恢复、重构
   - **系统分析员和数据库设计员**：需求分析与总体设计
   - **应用程序员**：基于外模式开发应用程序
   - **最终用户**：通过应用系统使用数据库

---

## 1.5 数据库系统的体系结构

1. **集中式数据库系统**：所有组件在一台计算机上
2. **客户-服务器数据库系统**：两层（C/S）→ 三层（浏览器/应用服务器/数据库服务器）
3. **并行数据库系统**：并行计算机上运行，高性能、高可用、高扩展
4. **分布式数据库系统**：物理分布在不同节点，逻辑上是一个整体（场地自治）
5. **云数据库系统**：部署在云计算环境，以服务形式提供数据库功能，能动态伸缩、按需分配

## 核心概念速记

| 概念 | 一句话解释 |
|------|-----------|
| 数据 | 描述事物的符号记录 |
| 数据库 | 有组织、可共享的数据集合 |
| DBMS | 管理数据的系统软件 |
| DBS | DB + DBMS + 应用 + DBA |
| 数据模型三要素 | 数据结构 + 数据操纵 + 完整性约束 |
| 三级模式 | 外模式 → 模式 → 内模式 |
| 两级映像 | 外模式/模式映像 + 模式/内模式映像 |
| 物理独立性 | 存储结构改变不影响应用程序 |
| 逻辑独立性 | 逻辑结构改变不影响应用程序 |
